Duck Hunt (NES) — Ghidra Setup Instructions
=============================================

ROM: Duck Hunt # NES.NES
iNES Header: 2 PRG banks (32KB), 1 CHR bank (8KB), Mapper 0 (NROM)

NROM = no bank switching. Single 32KB PRG mapped straight to $8000-$FFFF.

=== ROM FILE LAYOUT ===

Offset 0x0000 - 0x000F : iNES header (16 bytes, skip this)
Offset 0x0010 - 0x800F : PRG ROM (32,768 bytes = 32KB)
Offset 0x8010 - 0xA00F : CHR ROM (8,192 bytes = 8KB, graphics — not code)

=== MEMORY MAPPING FOR GHIDRA ===

PRG ROM maps to CPU address space $8000-$FFFF.

Import as RAW BINARY, language: 6502:LE:16:default

  File Offset   Size      CPU Address   Name
  ----------    ------    -----------   ----
  0x0010        0x8000    $8000         PRG_ROM

That's it. One block. NROM has no bank switching.

Do NOT import CHR ROM as code. It is tile graphics data.

=== VECTORS (last 6 bytes of PRG, at CPU $FFFA-$FFFF) ===

File offset 0x800A-0x800F (end of PRG ROM):

  $FFFA-$FFFB : NMI    = read from ROM
  $FFFC-$FFFD : RESET  = read from ROM
  $FFFE-$FFFF : IRQ    = read from ROM

After import, go to $FFFA and define these as word (2-byte) pointers.
Mark each target as a function entry point.

=== GHIDRA MCP SERVER ===

Start the Ghidra MCP bridge on port 8888.
Config already written to .mcp.json in this project directory.

=== STRICT RULES ===

After import:
- Mark ONLY the three vector targets (NMI, RESET, IRQ) as functions
- Do NOT run auto-analysis aggressively
- Do NOT let Ghidra speculatively create functions
- Unknown bytes = data until proven otherwise
- Claude will use the MCP server to query Ghidra — no manual analysis needed beyond this setup
